WHAT IT IS
--------------------------------------------------------------------------------

Capability Access Control List (CACL) is a FreeBSD kernel module that
controls which processes can use a file descriptor. A descriptor with
a CACL works normally for listed processes and returns EACCES for
everyone else.

  /dev/cacl — a character device for managing CACLs. Ioctls add and
  remove processes from a descriptor's access list. The device fd can
  be passed to other processes with Capsicum ioctl rights controlling
  what they can do.

  MACF policy — use-time enforcement. Every operation on a descriptor
  with a CACL checks the caller against the access list.

Both parts live in the same kernel module and share state.


FEATURES
--------------------------------------------------------------------------------

1. Identity

   Every process gets a unique random 64-bit token inside the kernel.
   Generated by arc4random(9). Never exposed to userspace. Processes
   refer to each other using process descriptors (pdfork(2)). Token
   changes on exec. Token copies on fork.

   Tokens are purely internal. The access list stores tokens. The
   use-time check compares tokens. No external identifiers (pids) are
   ever used for enforcement.


2. Access Lists

   A process opens /dev/cacl and issues ioctls to manage who can use
   a descriptor. Each descriptor has one access list. The list holds
   process tokens. At use-time, the kernel checks the caller's process
   token against the list — if it matches, the operation is allowed.

     CACL_IOC_ADD          Add processes by descriptor. Creates the
                           access list if none exists.
     CACL_IOC_ADD_SELF     Add the caller's process token. Creates
                           the list if none exists. No proc descriptor
                           needed.
     CACL_IOC_REMOVE       Remove processes by descriptor.
     CACL_IOC_CLEAR        Empty the entire access list.

   All operations are batched: arrays of cap fds and proc fds.


3. Delegated Authority via Capsicum

   /dev/cacl is a regular fd. Capsicum's cap_ioctls_limit(2) restricts
   which ioctls are allowed on it.

     cap_ioctl_t self_only[] = { CACL_IOC_ADD_SELF };
     cap_ioctls_limit(cacl_fd, self_only, 1);

   Pass the limited fd to a process. It can add itself to access
   lists but cannot add others, remove anyone, or clear lists. Pass
   it further — rights only narrow.

   Authority is the device fd. What you can do is controlled by
   which ioctls Capsicum allows.


4. Use-Time Enforcement

   Every operation on a descriptor with a CACL checks the caller's
   process token against the list. If it matches, allow. If not,
   EACCES. Descriptors without a CACL skip the check entirely.


INTERFACE
--------------------------------------------------------------------------------

All management goes through ioctls on /dev/cacl.

CACL_IOC_ADD           Add processes to access lists. Creates the
                       list if none exists.

CACL_IOC_REMOVE        Remove processes from access lists.

    struct cacl_members {
        int        *cm_cap_fds;       /* which descriptors          */
        uint16_t    cm_cap_count;
        int        *cm_proc_fds;      /* which processes            */
        uint16_t    cm_proc_count;
    };

CACL_IOC_ADD_SELF      Add the caller to access lists. Creates the
                       list if none exists. The kernel reads the
                       caller's process token directly from
                       curthread->td_ucred. No descriptor needed for
                       the identity — just cap fds for what to join.

CACL_IOC_CLEAR         Empty access lists.

    struct cacl_fds {
        int        *cf_cap_fds;       /* which descriptors          */
        uint16_t    cf_cap_count;
    };

    CACL_IOC_ADD_SELF and CACL_IOC_CLEAR both use cacl_fds.


IMPLEMENTATION
--------------------------------------------------------------------------------

One kernel module, two subsystems:

  make_dev() creates /dev/cacl at module load. The ioctl handler
  resolves cap fds and proc fds via fget(9), accesses MACF labels
  on the underlying objects, and stamps or modifies access lists.

  MAC_POLICY_SET() registers the MACF policy. Credential hooks assign
  process tokens. Use-time hooks enforce access lists.

Credential hooks:
    mpo_cred_init_label       Allocate label storage.
    mpo_cred_destroy_label    Free label storage.
    mpo_cred_create_init      Assign token to init process.
    mpo_cred_create_swapper   Assign token to swapper process.
    mpo_cred_copy_label       Copy token on fork.
    mpo_cred_execve_will      Assign new token on exec.

Object hooks (vnode, socket, pipe, posixshm, posixsem):
    mpo_*_init_label          Allocate access list storage.
    mpo_*_destroy_label       Free access list storage.

Use-time hooks:
    mpo_vnode_check_read, _write, _stat, _open, _poll
    mpo_socket_check_send, _receive, _connect, _bind,
        _listen, _accept, _poll, _stat
    mpo_pipe_check_read, _write, _ioctl, _poll, _stat
    mpo_posixshm_check_read, _write, _mmap, _truncate, _stat
    mpo_posixsem_check_post, _wait, _getvalue

Non-MACF kernel APIs:
    make_dev(9) / destroy_dev(9)
    arc4random(9)
    pdfork(2) / procdesc(9) — process descriptors
    fget(9)
    cap_ioctls_limit(2)


PROPERTIES
--------------------------------------------------------------------------------

Default-allow. Descriptors without a CACL are unaffected.
No breakage. SCM_RIGHTS is never blocked.
Tokens are hidden. Never exposed to userspace.
Access lists are permanent. Cannot be removed, only cleared.
Lists die with the object. Last close destroys the list.
Closing /dev/cacl freezes management. Enforcement continues.
Authority is the device fd. Capsicum controls what you can do.
Self-service. A process can add itself to access lists.
Batched. One ioctl for multiple descriptors and processes.
Composable. Capsicum for delegation, MACF for enforcement.
Loadable. Standard kmod. No kernel patches.


KNOWN LIMITATIONS
--------------------------------------------------------------------------------

Fork inherits the parent's token. A forked child can use the parent's
listed descriptors.
